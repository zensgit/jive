# 新增交易页改造计划：兼容 Transaction Editor 与 OneTouch

## 1. 背景

当前 Jive 的新增交易页集中在 `lib/feature/transactions/add_transaction_screen.dart`，能力已经非常完整：

- 支出 / 收入 / 转账
- 分类、账户、标签、项目、预算排除
- 语音输入与语义解析
- 跨币种转账
- 商户记忆
- 编辑已有交易 / 基于已有交易预填

但页面也因此承担了过多职责：

- 既想做“完整交易编辑器”
- 又承接“快速记一笔”
- 还承担“语音 / OCR / 自动草稿 / 快捷指令”的确认入口

这会带来两个问题：

1. 普通手动记账时，页面信息密度过高，首屏决策成本偏高。
2. 快速入口虽然已经存在，但没有形成统一的 OneTouch 式体验，无法做到“尽量直接提交，不完整时自然退回编辑器”。

本计划目标是在不推翻现有能力的前提下，把当前新增交易页改造成：

- 默认是轻量、聚焦、可快速提交的录入页
- 同时又是完整、可靠、可兜底的交易编辑器
- 并且能承接未来的 OneTouch / 闪记动作 / 小组件 / 快捷指令 / 自动草稿

## 2. 改造目标

### 2.1 产品目标

- 将“新增交易页”统一为所有交易输入入口的最终确认与编辑页面。
- 默认首屏只暴露核心字段，适配日常高频录入。
- 借鉴 OneTouch 思路，支持“预填 + 直接提交 / 轻确认 / 打开编辑器补全”。
- 让语音、模板、快捷指令、自动草稿、导入预览等来源都能复用同一套交互与字段。

### 2.2 交互目标

- 手动新增交易时，用户应能在首屏完成大多数录入。
- 来源明确时，页面顶部应告诉用户“这笔交易是怎么来的、为什么会被这样预填”。
- 高级能力不丢失，但默认收起，只在需要时展开。
- 对“快速动作”提供沉淀路径，让用户能把高频交易保存为可复用动作。

### 2.3 工程目标

- 尽量复用现有状态与服务层能力。
- 先做页面结构与交互重组，再逐步补 OneTouch 动作模型。
- 保持已有编辑交易、预填交易、语音入口兼容。

## 3. 设计原则

### 3.1 一页两用

同一个页面同时承担：

- `Quick Entry`：快速录入
- `Transaction Editor`：完整编辑

差别不靠两个完全独立页面实现，而靠：

- 不同来源的预填
- 默认展开层级不同
- 提交动作不同

### 3.2 核心字段前置

首屏只聚焦以下字段：

- 金额
- 交易类型
- 时间
- 场景/账本范围（现阶段至少是账本）
- 账户
- 分类
- 备注

### 3.3 高级字段后置

以下字段默认收起到“高级选项”：

- 标签
- 项目
- 预算排除
- 转入账户 / 汇率 / 转账高级设置
- 附件
- 规则解释 / 智能标签解释

### 3.4 来源驱动

页面要显式区分以下来源：

- 手动新增
- 模板 / 快速动作
- 语音
- 快捷指令 / 深链
- 自动草稿
- OCR / 导入预览
- 编辑已有交易

不同来源决定：

- 顶部提示文案
- 默认展开内容
- 提交按钮文案
- 是否允许直接提交

### 3.5 直接提交优先，编辑兜底

借鉴 OneTouch：

- 如果信息完整，优先允许直接提交
- 如果信息部分完整，先轻确认
- 如果信息不完整，自动进入编辑补全

## 4. 目标交互模型

## 4.1 页面模式

新增交易页统一为一个页面，内部支持 4 种模式：

### A. 手动模式

入口：

- 首页 FAB
- 首页“记一笔”

特点：

- 不显示来源横幅
- 默认展开核心字段
- 高级字段收起

### B. 快速动作模式

入口：

- 模板快记
- 未来的 OneTouch / 闪记动作

特点：

- 顶部显示“来自快速动作 xxx”
- 若动作信息完整，可显示“立即保存”主按钮
- 若动作信息不完整，进入预填编辑

### C. 草稿确认模式

入口：

- 自动记账草稿
- 导入预览后的单条修订
- OCR / 语音结果确认

特点：

- 顶部显示来源解释
- 首屏突出“待确认字段”
- 主按钮使用“确认入账”

### D. 编辑模式

入口：

- 修改已有交易

特点：

- 标题为“编辑交易”
- 显示原始来源、智能标签、关联信息
- 主按钮为“保存修改”

## 4.2 页面结构

建议的页面结构如下：

1. 顶部导航区
2. 来源横幅区
3. 金额与类型区
4. 核心字段区
5. 快速动作推荐区
6. 高级选项折叠区
7. 底部操作区

### 4.2.1 顶部导航区

信息：

- 返回
- 标题：`记一笔` / `确认交易` / `编辑交易`
- 可选操作：更多、删除、复制

### 4.2.2 来源横幅区

仅在非手动模式显示。

文案示例：

- 来自快速动作“午餐”
- 来自自动识别，请确认分类与账户
- 来自语音输入，已帮你解析金额与时间
- 来自快捷指令，已预填主要字段

作用：

- 告诉用户为什么页面是当前状态
- 降低“不知道系统改了什么”的疑惑

### 4.2.3 金额与类型区

这是页面第一焦点。

组成：

- 大金额显示与数字键盘
- 支出 / 收入 / 转账三段切换
- 常用时间快捷入口：今天 / 昨天 / 自定义

要求：

- 金额必须永远位于首屏最显眼位置
- 类型切换尽量轻量，不和其他字段竞争

### 4.2.4 核心字段区

必须首屏可见：

- 账本 / 场景
- 账户
- 分类
- 时间
- 备注

设计要求：

- 账户、分类为块级可点击卡片，而不是拥挤的表单行
- 备注支持高频短语建议
- 如果来源已预填，对应项可展示“已预填”态

### 4.2.5 快速动作推荐区

位于核心字段下方，展示：

- 最近使用动作
- 与当前分类 / 场景相关的推荐动作
- “保存为快速动作”入口

目标：

- 从编辑页反向沉淀 OneTouch 能力

### 4.2.6 高级选项折叠区

使用折叠卡或展开分组，收纳：

- 标签
- 项目
- 转账高级设置
- 多币种
- 不计入预算
- 附件
- 智能标签解释
- 原始文本 / 来源信息

要求：

- 默认收起
- 一旦用户展开，页面记住偏好

### 4.2.7 底部操作区

底部固定主操作，避免页面滚到底才能提交。

按钮策略：

- 主按钮：`保存` / `确认入账` / `立即记录`
- 次按钮：`存为快速动作`
- 可选按钮：`继续新增`

注意：

- 不要同时放 3 个同权主 CTA
- 主按钮文案应由来源模式驱动

## 5. OneTouch 在 Jive 中的产品落地

## 5.1 命名建议

建议不直接使用 `OneTouch` 作为最终中文产品名，可采用：

- 闪记动作
- 快捷动作

本文用“快速动作”代称。

## 5.2 快速动作定义

快速动作是“可复用的交易预设”。

最小字段集建议：

- 名称
- 图标 / 颜色
- 交易类型
- 默认账本 / 场景
- 默认账户
- 默认分类
- 默认标签
- 默认金额（可空）
- 默认备注（可空）
- 是否允许直接提交
- 是否显示在首页

## 5.3 快速动作的三种执行方式

### A. 直接提交型

适用：

- 早餐 12
- 地铁 3
- 停车费 5

规则：

- 字段齐全
- 点一下直接入账
- 成功后短暂 toast

### B. 轻确认型

适用：

- 午餐
- 打车
- 买菜

规则：

- 已有账户 / 分类 / 类型
- 只缺金额或备注
- 点击后进入轻量确认编辑器

### C. 复杂编辑型

适用：

- 信用卡还款
- 报销到账
- 跨币种转账
- 朋友代付 / 借还

规则：

- 点击后打开完整编辑器
- 但页面字段已高度预填

## 5.4 与现有模板的关系

当前 `JiveTemplate` 已具备部分基础，可先采用“模板升级为快速动作”的兼容策略。

阶段性策略：

- 第一阶段：模板 UI 升级，作为快速动作承载
- 第二阶段：补充直接提交、展示位置、来源横幅等能力
- 第三阶段：如有必要，再拆出独立模型

## 7. 分阶段实施计划

## 阶段 1：页面结构重组

目标：

- 不改底层数据模型
- 先把当前页面从“大表单”重组成“核心字段 + 高级折叠”

改动项：

- 提炼来源横幅组件
- 提炼核心字段区组件
- 将标签 / 项目 / 预算排除 / 汇率等移入高级区
- 底部改为固定提交区
- 为不同来源切换按钮文案

验收标准：

- 手动新增时首屏只看到金额、类型、时间、账户、分类、备注
- 编辑功能不丢失
- 转账与多币种流程仍可完成

## 阶段 2：快速动作接入

目标：

- 让模板条与新增交易页形成统一的预填链路

改动项：

- 在新增交易页增加“来自快速动作”的来源态
- 支持从页面将当前填写结果保存为快速动作
- 快速动作支持直接提交或打开编辑器
- 首页模板条升级为“快速动作条”

验收标准：

- 至少支持 1 个完整动作直接入账
- 至少支持 1 个半预填动作进入编辑器
- 动作执行结果有清晰反馈

## 阶段 3：来源统一

目标：

- 让语音、快捷指令、自动草稿、导入预览都进入同一套编辑器逻辑

改动项：

- 为不同来源补充统一的页面入参结构
- 页面顶部显示来源说明
- 草稿确认时主按钮统一为“确认入账”
- 对不完整来源统一高亮待补字段

验收标准：

- 用户可从页面一眼看出来源
- 不同入口不再需要独立确认样式

## 阶段 4：系统级入口联动

目标：

- 为后续小组件 / App Intent / 快捷指令铺路

改动项：

- 新增“可直接提交”的快速动作语义
- 为 iOS 深链与未来 App Intent 统一字段映射
- 为小组件动作保留动作 ID 与预填协议

验收标准：

- 从外部入口跳转时，页面可稳定识别来源与预填字段

## 8. 推荐的数据与参数调整

## 8.1 页面入参建议

为 `AddTransactionScreen` 引入更明确的来源参数，例如：

- `entrySource`
- `sourceLabel`
- `canDirectSubmit`
- `highlightMissingFields`
- `quickActionId`

目的：

- 避免页面仅依赖零散布尔值推断当前模式

## 8.2 快速动作模型建议

若后续模板不够用，可新增独立模型：

- `id`
- `name`
- `icon`
- `color`
- `type`
- `bookId`
- `sceneId`（未来）
- `accountId`
- `categoryKey`
- `tagKeys`
- `defaultAmount`
- `defaultNote`
- `submitMode`：direct / confirm / edit
- `showOnHome`
- `showInShortcuts`
- `usageCount`

## 9. 视觉与文案建议

## 9.1 页面标题

- 手动新增：`记一笔`
- 快速动作：`快速记录`
- 草稿确认：`确认交易`
- 编辑：`编辑交易`

## 9.2 提交按钮

- 手动新增：`保存`
- 快速动作完整：`立即记录`
- 草稿确认：`确认入账`
- 编辑：`保存修改`

## 9.3 空状态与提示

- 来源横幅要短，不写成长段说明
- 高级选项默认只展示摘要，不占首屏
- 对自动识别要强调“已帮你预填”，不是“系统替你决定”

## 10. 风险与注意事项

### 10.1 功能回归风险

当前页面能力很多，重组时最容易回归：

- 转账
- 跨币种
- 编辑已有交易
- 预算排除
- 标签与项目
- 语音起录后的回填

### 10.2 状态复杂度风险

当前 `AddTransactionScreen` 已经较大，直接在原文件继续堆逻辑会增加维护成本。

建议：

- 页面主状态暂时保留
- 但 UI 组件尽快拆分
- 第二阶段后再考虑控制器进一步拆分

### 10.3 模型边界风险

模板与 OneTouch/快速动作边界如果不明确，容易出现两个概念并行。

建议：

- 短期把模板当快速动作载体
- 长期再决定是否拆模

## 11. 验收清单

### 11.1 交互验收

- 首屏是否只暴露核心字段
- 是否能从来源横幅识别当前上下文
- 是否能在 3 秒内完成一笔常见支出
- 是否能把当前交易保存为快速动作
- 快速动作是否支持直接提交与编辑两种路径

### 11.2 工程验收

- 页面无明显分析告警
- 现有测试至少不因基础改造大面积失效
- 编辑已有交易流程保持可用
- 导入 / 自动草稿 / 语音入口仍可进入该页

### 11.3 用户体验验收

- 手动新增不再像“填表”
- 高级能力仍找得到
- 用户知道系统帮他做了什么
- 高频行为能自然沉淀为快速动作

## 12. 建议的实际执行顺序

如果按最稳妥路径推进，建议顺序如下：

1. 重构 `AddTransactionScreen` 的 UI 结构，不动数据模型。
2. 引入来源横幅、底部固定提交区、高级选项折叠。
3. 将模板条升级为快速动作条，并接入新增交易页的来源模式。
4. 增加“保存为快速动作”能力。
5. 统一语音 / 自动草稿 / 快捷指令的入参表达。
6. 最后再考虑小组件与系统快捷方式的更深整合。

---

本计划优先服务于当前页面的可执行改造，不要求先完成“场景层”或更大范围 IA 重构；但页面结构已为后续的场景、快速动作、小组件、快捷指令统一入口预留空间。
